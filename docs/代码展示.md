# CypherGuard AI：功能模块、代码导读与调试说明

本文按**功能点**归纳：每个功能说明「做什么」「代码在哪」「如何调试」。适用于阅读源码、联调与排障。

---

## 一、系统分层速览

| 层级 | 说明 | 关键位置 |
|------|------|----------|
| 入口网关 | HTTP 反向代理，路由到各服务；SSE 流式需关闭缓冲 | `gateway/nginx.conf` |
| 前端 | React + Vite，单页在 `/dashboard`，侧栏切换「知识库 / 智能对话」等 | `frontend/src/` |
| 业务服务 | FastAPI 微服务，路径前缀与网关一致 | `services/*/app/main.py` |
| 异步 Worker | Celery：文档解析、摘要、向量化写入 Qdrant | `services/worker/app/celery_app.py` |
| 共享库 | 模型、Schema、数据库访问 | `shared/python/common/` |
| 基础设施 | Postgres、Redis、MinIO、Qdrant | `docker-compose.yml` |

**本地调试入口**

- 网关（经 Docker）：`http://localhost`（80）→ 各服务 `/auth`、`/kb`、`/chat`、`/models`
- 前端开发服务器：`http://localhost:5173`（compose 中 `frontend` 服务）
- 环境变量：根目录 `.env`；前端 `frontend/.env` 中 `VITE_API_BASE_URL` 指向网关或直连服务

## 二、按功能点归纳

### 1. 认证与用户（Auth）

**功能说明**

- 注册、登录、JWT 访问令牌与刷新、`/auth/me` 拉取当前用户、个人资料与改密。

**核心代码**

| 角色 | 路径 |
|------|------|
| 后端路由 | `services/auth-service/app/main.py`（`/auth/register`、`/auth/login`、`/auth/refresh`、`/auth/me` 等） |
| 前端状态与请求 | `frontend/src/contexts/AuthContext.tsx`、`frontend/src/pages/Login.tsx` |
| HTTP 封装（含 401 刷新） | `frontend/src/services/api.ts` 顶部拦截器 |
| 路由保护 | `frontend/src/components/ProtectedRoute.tsx` |

**调试建议**

- 浏览器：Application → Local Storage 查看 `access_token` / `refresh_token`
- 后端：对 `auth-service` 容器日志 `docker logs cypherguard-auth-service`，或在本地以 `POSTGRES_URL`、`REDIS_URL`、`JWT_SECRET` 运行该服务后断点 `main.py`
- 快速探测：`curl http://localhost/auth/healthz`（经网关；若网关未暴露则 `http://localhost:8000/healthz` 直连容器内）

### 2. 知识库（KB）— 列表、详情、搜索

**功能说明**

- 知识库 CRUD、按知识库搜索（chunk / FAQ）、文档列表与元数据。

**核心代码**

| 角色 | 路径 |
|------|------|
| KB API | `services/kb-service/app/main.py`（`/kb/knowledge-bases`、`/kb/knowledge-bases/{id}/search`） |
| 前端页面 | `frontend/src/pages/KnowledgeBase.tsx` |
| API 方法 | `frontend/src/services/api.ts` 中 `kbAPI` 相关 |

**调试建议**

- 确认 JWT：请求头带 `Authorization: Bearer <token>`
- 数据库：文档与 KB 元数据在 Postgres（见 `shared/python/common/models.py`）
- 搜索若异常：同时查 `kb-service` 日志与 Worker 是否已向 Qdrant 写入向量（见功能点 4）

### 3. 文档生命周期 — 上传、导入 URL、任务状态

**功能说明**

- 上传文件至 MinIO、URL 导入、触发解析与「学习/重索引」；任务列表反映 Celery 进度。

**核心代码**

| 角色 | 路径 |
|------|------|
| 上传 / URL / 任务 API | `services/kb-service/app/main.py`（`/kb/documents/upload`、`import-url`、`/kb/tasks` 等） |
| 异步执行 | `services/worker/app/celery_app.py`（解析文本、分块、embedding、写 Qdrant） |
| 文档详情 UI | `frontend/src/pages/DocumentDetail.tsx` |
| 状态枚举 | `shared/python/common/document_status.py` |

**调试建议**

- **队列不消费**：检查 `redis` 是否可达、`cypherguard-worker` 是否运行、`docker logs cypherguard-worker`
- **长期 `processing`**：在 `celery_app.py` 中任务函数打断点或加日志，确认 MinIO 下载、`extract_text`、调用 `model-gateway` embedding 是否成功
- **MinIO**：控制台端口 compose 中为 `9001`（见 `docker-compose.yml`），核对 bucket 与 `source_uri`（`s3://`）

---

### 4. 向量化与检索（Worker + Qdrant + Model Gateway）

**功能说明**

- Worker 将 chunk 向量化，写入 Qdrant collection（如 `kb_chunks_v2`）；检索时由 chat 或 kb 侧发起向量查询；向量与聊天补全由 **Model Gateway** 对内提供。

**核心代码**

| 角色 | 路径 |
|------|------|
| Qdrant 写入与 collection | `services/worker/app/celery_app.py`（`ensure_collection_exists`、`COLLECTION_NAME`） |
| 对内 LLM / 向量 / 重排 | `services/model-gateway/app/main.py`（`/internal/embeddings`、`/internal/chat`、`/internal/rerank`、`/internal/chat/stream`） |
| Chat 服务调用网关 | `services/chat-service/app/main.py`（检索与生成逻辑，搜索 `MODEL_GATEWAY_URL`） |

**调试建议**

- Qdrant：浏览器或 REST `http://localhost:6333`（compose 映射）查看 collection 与 points
- Model Gateway：先 `GET /healthz`，再对 `/internal/embeddings` 发小样本 JSON 验证维度与模型配置
- 维度不一致：Worker 建 collection 的 `vector_size` 需与 embedding 模型输出一致

---

### 5. 智能对话与 RAG（会话、消息、流式）

**功能说明**

- 会话 CRUD、消息非流式与 **SSE 流式**、可选 Agent 路径；结合知识库检索与引用展示。

**核心代码**

| 角色 | 路径 |
|------|------|
| Chat API | `services/chat-service/app/main.py`（`/chat/conversations`、`messages`、`messages/stream`、`messages/agent`） |
| 网关 SSE | `gateway/nginx.conf` 中 `location ~ ^/chat/conversations/.+/messages/stream$`（`proxy_buffering off`） |
| 前端对话页 | `frontend/src/pages/Chat.tsx`、`frontend/src/pages/Dashboard.tsx`（会话列表与模式） |
| 消息渲染 / 引用 | `frontend/src/components/chat/*`（如 `AnswerMessage.tsx`、`AnswerReferences.tsx`、`MarkdownRenderer.tsx`） |
| 前端 API | `frontend/src/services/api.ts` 中 `chatAPI` |

**调试建议**

- **流式无输出**：查网关是否走 `messages/stream` 路径、Nginx 是否禁用缓冲；浏览器 Network → EventStream
- **非流式正常、流式 502**：多为网关超时，可调 `proxy_read_timeout`（已设 300s）
- 后端：在 `main.py` 的 stream 生成器处打断点，确认检索结果与 model-gateway 流式响应

### 6. 对话模式（Modes）

**功能说明**

- 预设对话模式（system prompt、top_k、是否强制引用等），影响 RAG 行为。

**核心代码**

- API：`services/chat-service/app/main.py`（`/chat/modes` CRUD）
- UI：`Dashboard` / `Chat` 中与 `selectedMode` 相关的状态（见 `Dashboard.tsx`）

**调试建议**

- 用 `GET /chat/modes` 核对数据库中的模式是否与 UI 选择一致

---

### 7. 模型配置（管理员向）

**功能说明**

- 多模型配置、默认模型、连通性测试（含 Ollama）。

**核心代码**

| 角色 | 路径 |
|------|------|
| API | `services/model-config-service/app/main.py`（`/models` CRUD、`set-default`、`test`、`test-ollama-connection`） |
| 网关 | `gateway/nginx.conf` → `location /models` |
| 前端设置页 | `frontend/src/pages/Settings.tsx` 及 `frontend/src/components/settings/*` |
| 前端 API | `frontend/src/services/api.ts` 中 `settingsAPI` |

**调试建议**

- 模型列表为空：查 `model-config-service` 与 Postgres 中模型表
- 测试连接失败：核对 `.env` 中各厂商 API Key / Ollama 地址是否可从 **model-config 容器** 内访问

### 8. FAQ、标签、分块与实体

**功能说明**

- FAQ CRUD/导入、标签管理、文档分块编辑、实体列表（用于增强检索或展示）。

**核心代码**

- 后端：`services/kb-service/app/main.py`（`/kb/faq`、`/kb/tags`、`/kb/documents/{id}/chunks`、`entities`）
- 前端：`frontend/src/pages/FAQManager.tsx`、`TagManager.tsx`；文档详情中的 chunk 相关在 `DocumentDetail.tsx`

**调试建议**

- FAQ 导入格式错误：看 `kb-service` 返回的 4xx 消息与请求体 schema（`shared/python/common/schemas.py`）

## 四、阅读代码的推荐顺序

1. `gateway/nginx.conf` — 弄清 URL 如何转发  
2. `frontend/src/services/api.ts` — 前端与后端契约  
3. `services/*/app/main.py` — 按业务线读路由  
4. `services/worker/app/celery_app.py` — 文档入库全链路  
5. `shared/python/common/` — 数据模型与配置  

按上述顺序即可从「请求从哪进」一直跟到「数据落哪、模型怎么调」。
